## 关键风险点

| 风险 | 影响度 | 概率 | 缓解方案 |
|-----|--------|------|--------|
| 后端 API 延期 | 🔴 高 | 高 | 使用 Mock Server 继续开发 |
| 硬件协议复杂 | 🔴 高 | 中 | 早期与硬件方沟通、预留时间 |
| 第三方支付审核 | 🟡 中 | 中 | 提前启动申请、预留 1-2 周 |
| 测试覆盖不足 | 🟡 中 | 中 | 建立自动化测试文化、预留时间 |
| 安全漏洞 | 🔴 高 | 低 | 代码审查、安全审计 |

## 团队配置建议

### 最小化团队（MVP 阶段）
```
- FrontEnd Developer: 2-3 人
  └─ 1 人主导 UI 和交互
  └─ 1 人负责 API 集成和状态管理
  └─ 1 人（可选）后续添加

- BackEnd Developer: 1 人
  └─ 负责 API 服务搭建
  └─ 数据库设计
  └─ 认证系统

- DevOps/QA: 0.5 人
  └─ 部分时间参与
```

### 标准配置（功能完善阶段）
```
- FrontEnd: 3 人
- BackEnd: 2 人  
- DevOps: 1 人
- QA/测试: 1 人
- PM/UX: 1 人

总计: 8 人开发团队
```

## 质量检查清单

### 上线前的必检项
- [ ] 所有页面可导航（无 404）
- [ ] 用户认证流程完整（登录→首页→登出）
- [ ] 核心业务流程可完成（场景生成、会话、订阅）
- [ ] 没有 JS 错误和异常（控制台清洁）
- [ ] 性能指标达标（首屏 < 3s, Lighthouse > 50)
- [ ] 安全基础（无 XSS、CSRF、敏感数据泄露）
- [ ] 适配测试（iOS Safari、Android Chrome）
- [ ] 回归测试（主流程手工测试）

### MVP 交付标准
```
功能完整度:  60-70%
Bug severity: 1-2 个 Critical, < 5 个 High
测试覆盖:    单元 > 40%, E2E > 3 个主流程
性能指标:    FCP < 2s, LCP < 4s
安全评分:    无 Critical / High 漏洞
```
